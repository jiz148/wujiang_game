## 1. 走查方法

本次使用与玩家相同的 HTTP 战役入口和权限链，执行一名真人席位与主要 AI 势力的完整确定性走查。战役固定为 8 城、2 个主要势力、6 个中立城邦和 12 个战略月；没有修改数据库、跳月或人工修复世界状态。

自动化玩家承担主公职位，所有治理、事件、科技、祭祀、冲突、月度推进和归档都通过公开 API 与合法官职执行。城市冲突采用 `quick`，因此本次证明战略冲突闭环，但不作为真实格子战观看体验验证。

## 4. 走查中发现并修复的问题

### 同一接收者的职位命令错误互相覆盖

原实现使用“已落地职位命令数 + 1”生成队列键，没有计入本月尚未结算的职位命令。主公同月向同一城主下达“设置方针”和“处理事件”两个不同目标时，第二条会被误判为同键替换。

现已把本月待结算的命令/请求数量计入 action key。不同命令可以分别消耗军令、分别执行；真正同键的普通城市方针等仍保持原替换语义。完整走查同时覆盖两条命令均保留和前三月治理目标完成。

## 6. 产品观察与后续边界

- 本次 12 月只有一场冲突且没有城市易主，说明闭环已经成立，但中立城邦仍偏向静态资源点。和平整合、关系、保护和占领政策应继续留在 Phase 3，而不是回填到 Phase 2。
- 每个主要势力几乎每月都有一个事件，双方累计 21 次选择。当前事件仍可读且能由职位链自动处理，但真实玩家测试应继续观察是否产生节奏疲劳。
- 本次只验证快速城市战。手动/观看/AI 格子战已有独立行为覆盖，但完整 60～90 分钟真人战役的实际耗时、理解成本和格子战插入体验仍没有真实玩家样本，不能写成已验证指标。
- Phase 2 的工程范围和自动化退出门槛已经满足；是否接受上述未验证风险并进入 Phase 3，必须由项目负责人明确确认。
