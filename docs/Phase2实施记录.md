# Phase 2 实施记录

> 当前状态：进行中  
> 进入日期：2026-07-17  
> 阶段目标：把现有战略沙盒收束为有时限、有目标、有结算、可以继续自由游玩的完整大战役。

## 已完成基础设施：原路线 P2.1 账号武将熟练度与下一目标

P2.1 复用账号已完成战绩生成只读成长档案，不增加攻击、防御、生命、资源或其他战斗数值。熟练度只表达玩家对某名武将的实际使用经验。

### 数据规则

- 每场已完成且归属于当前账号的对局，对该账号实际控制席位阵容中的每个不同武将增加 1 点熟练度；同场选择多个同名单位仍只计一次。
- 胜利时，本场使用的每个不同武将额外增加 1 点熟练度。
- 熟练等级统一为：`初识 1`、`熟练 3`、`精通 8`、`专家 15`、`大师 30`。达到大师后不再显示虚构的下一等级，但继续累计出战、胜场和熟练度点数。
- 聚合结果包含账号总完成场次、胜场、胜率，以及每名武将的出战、胜场、负场、熟练度点数、当前等级、下一等级和剩余点数。
- 下一目标优先选择距离下一等级最近、熟练度最高、最近使用时间最新的武将；没有战绩时引导完成第一场正式对局。
- 熟练度从现有账号战绩派生，不建立可与战绩不一致的第二套可写计数器；旧战绩自动参与计算。

### 玩家路径

- 登录后的首页账号记录区先显示熟练度总览、最接近升级的下一目标和常用武将，再显示最近战绩。
- 对局结束且战绩完成保存后，赛后总结显示更新后的下一熟练目标。
- 空战绩、读取中、读取失败和登出状态必须有明确反馈；熟练度接口只返回当前登录账号的数据。

### 后续边界

- P2.1 不实现每日/每周刷新、不发放战斗数值奖励，也不把熟练度描述成武将强度或平衡评级。
- 英雄专属机制挑战、周期目标和平衡看板在后续 P2 子阶段接入同一账号战绩基础。

### 实现结果

P2.1 已完成账号私有 `/api/progression/overview`、战绩派生聚合、首页熟练度总览、最多四名常用武将进度和赛后下一目标。读取事件只记录匿名会话、来源与是否空状态，不记录账号、session、房间 token 或武将战绩明细。

行为覆盖包括：胜方武将获得 2 点、败方武将获得 1 点、账号之间数据隔离、未登录拒绝访问、不提供战斗数值，以及前端熟练度/目标渲染。

该项于 2026-07-17 完成。产品主线转向大战役后继续保留为支持基础，但不再作为当前 Phase 的编号或继续扩张方向。

## 当前路线 P2.1 战役契约

### 产品规则

- 公开创建入口固定生成“十二月城邦争衡”：8 城、2 个主要势力、6 个中立城邦、12 个战略月，预计 60～90 分钟。
- 统一全部城市或消灭其他主要势力仍会提前结算；未提前结束时，第 12 月完成城市收入、维护、叛乱和事件后进入十二月评议。
- 评议只排列主要势力：每城 100 分、所控城市平均民心 0～100 分、有城存续 100 分/流亡存续 25 分、每场城市战胜利 25 分且最多 100 分；世界主线未开放时为 0 分。
- 同分依次比较城市数、平均民心和城市战胜场，仍相同则并列。
- 结算后冻结推进、月度队列、城市方针、战术科技、进攻和防守英灵调整。房主可以保留结算并继续沙盒；继续后解除月份上限，但不重复结算。
- 旧存档没有 `campaign_contract`，继续按原长期沙盒读取，不会被强制设置为第 12 月结束。

### 实现结果

- `WorldState` 持久化 `campaign_contract` 与 `campaign_conclusion`，旧 JSON 缺少字段时保持兼容。
- `objectives.py` 统一生成限时排名、提前胜利/十二月评议、一次性结算事件、命令冻结和继续沙盒状态。
- 公开创建 API 不再接受旧客户端改变首个产品场景规模；内部 `StrategyStore.create_campaign` 仍保留无契约沙盒能力，供旧测试与后续场景工具使用。
- 战斗结算会立即检查提前胜利；月度结算会在第 12 月生成并持久化评议。
- 战情室显示场景构成、预计时长、当前/总月份、剩余月份、开放与未开放系统、逐项评分和房主继续沙盒入口。
- 后端与前端都在等待结算选择时阻止继续下令；继续后允许进入第 13 月及以后。

### 验证

- 战略单元测试覆盖第 12 月评议、排名字段、结算幂等、命令冻结、继续沙盒和存档往返。
- HTTP 行为测试覆盖固定场景、旧客户端参数不可改变契约、第 12 月结算、重复推进/下令拒绝、房主继续沙盒和第 13 月推进。
- 前端行为测试覆盖场景说明、逐项评分、推进按钮禁用与继续沙盒入口。

## 当前路线 P2.2 月初—月末信息闭环

### 产品规则

- 战情室每月固定回答三个问题：“上月发生了什么 / 本月必须处理什么 / 推进后预计发生什么”。首月明确显示尚无上月记录。
- 月结后保存结构化报告，包含城市归属、资源、民心、城防、叛军、执行计划和相关重大事件的变化；最多保留最近 24 份，旧存档缺少报告时兼容读取。
- 月结预测对己方城市复用正式结算中的方针收入、粮食维护、支持度和叛乱函数。排队后的最终城市方针参与预测；行动队列和建筑计划说明其在城市月结前执行及影响月份。
- 预测不计算或泄露战争结果、事件选择结果和 AI 决策，并在界面持续显示这一边界。
- 每次提交行动都明确反馈是新增还是替换，替换时返回原计划，同时显示剩余军令、当前势力资源余额和当前月/下月影响范围。
- 持久化原始月报不直接进入公开世界 JSON；客户端只接收按势力过滤后的月度决策切片。

### 实现结果

- `monthly_cycle.py` 统一负责城市确定性预测、月结差异报告、势力可见性过滤和三段式公开模型。
- `WorldState.monthly_reports` 保留月报并兼容旧存档；推进月份在玩家队列、AI 行动和正式月结全部完成后记录报告。
- `CampaignRecord.to_public_dict` 将当前行动队列叠加到各势力月度决策模型；原始报告由公开世界状态移除。
- `/api/strategy/campaigns/queue-action` 返回结构化 `submission`，前端据此显示新增/替换、军令、资源和影响月份。
- 战情室新增“月度决策”面板，显示城市资源净变化、粮食维护、民心变化、叛乱阶段和本月行动队列。

### 验证

- 战略单元测试覆盖预测与实际城市月结一致、月报存档往返、势力过滤和队列替换兼容。
- HTTP 行为测试覆盖月结后报告、公开状态不泄露原始月报，以及新增/替换响应的军令、资源和影响月份。
- 前端行为测试覆盖三段式标题、城市数字预测和行动队列说明。

## 当前路线 P2.3 前三个月引导

### 产品规则

- 引导只服务固定首场“十二月城邦争衡”，不附加到旧存档和普通沙盒。
- 第一月完成查看边境、设置城市方针和主动处理事件；第二月完成祭祀或任命；第三月按当前官职完成征募、编军、请调兵、防守、攻防命令或进攻中的一种冲突准备。
- 引导不锁月份、不更改行动权限、不消耗额外军令、不改变资源，也不强迫不同官职执行相同动作；目标可以提前完成，第三月后仍可补做。
- 当前合法行动队列提供即时完成反馈，月结后以成功事件、职位命令或战斗记录为永久证据；失效计划不会永久计入。
- 跳过按势力持久化，界面明确说明跳过不会获得/失去资源、不会替玩家下令、不会跳过战略月份；完成和跳过均无数值奖励。

### 实现结果

- `campaign_tutorial.py` 定义五个固定目标、月份章节、行动/世界事件完成证据、场景启用边界和跳过规则。
- `WorldState.campaign_tutorial` 只保存必须持久化的查看确认与跳过状态；公开战役状态移除原始进度字典，并按主要势力生成派生目标进度。
- `/api/strategy/campaigns/guide-action` 提供查看边境确认和跳过操作，复用战役成员权限并持久化事件日志。
- 战情室显示 `已完成/进行中/逾期可补做/后续月份` 状态、目标说明和按目标定位入口；完成或跳过后收束为紧凑状态提示。

### 验证

- 战略单元测试覆盖固定场景启用、五项目标派生、查看确认、跳过存档往返、资源不变和旧沙盒拒绝。
- HTTP 行为测试覆盖首场公开引导、查看边境、跳过月份、资源不变与内部进度不泄露。
- 前端行为测试覆盖三个月章节、五个目标、查看/跳过入口及无奖励说明。

## 当前路线 P2.4 官职自动化与决策降噪

### 产品规则

- 首场固定战役中，真人势力内的 AI 官职继续承担常规案牍；存在真人不再等于整个势力关闭 AI。
- 城市默认方针持续生效。AI 城主只在缺粮/叛乱危机、没有玩家方针计划且仍有剩余军令时自动调整一城，消耗 1 点剩余军令，不覆盖或挤掉玩家计划。
- 战情室每月只优先显示 1～3 项高后果决定：待决事件、城市生存危机和边境威胁；无危机时显示主动机会。
- 未收到明确命令时，AI 官职不自动宣战、不进入格子战、不替玩家选择事件结果。
- 命令和请求提交后显示发起/执行职位、军令成本与预计月份；月结后显示 `pending/accepted/completed/rejected` 状态和实际结果。
- 方针、事件和祭祀的结构化委托可以由有权限的 AI 接收者执行；攻防与一般自由文本命令只接受入案，不能把文本当成任意世界修改指令。
- 自动化仅用于固定首场契约，旧沙盒保持兼容。

### 实现结果

- `office_automation.py` 负责玩家势力内 AI 官职执行、剩余军令约束、命令模板回执、关键决定摘要和常规维护公开模型。
- 月度推进顺序扩展为：玩家队列 → 玩家势力 AI 官职委托/紧急维护 → 其他 AI 势力 → 城市月结。
- `OfficeOrder.details` 保存执行职位、预计完成月、解决月与结果摘要；原有序列化直接兼容旧订单。
- 公开战役增加按势力过滤的 `office_coordination`，包含最多三项关键决定、每城常规维护模式和最近八条命令/请求回执。
- 队列提交的 `submission.execution` 立即提供执行者、成本、预计回执月和交付说明；战情室与职位案牍显示最终结果。

### 验证

- 战略单元测试覆盖紧急方针、剩余军令为零、AI 职责完成、结构化委托实际执行、结果回执和旧沙盒不启用。
- HTTP 行为测试覆盖主公向 AI 城主下令、提交成本/预计月，以及推进后城市改变和完成回执。
- 前端行为测试覆盖 1～3 项关键决定、常规维护说明和命令回执的执行者/成本/预计月/结果。

P2.4 于 2026-07-18 完成。完整自动化回归 607 项通过，8 项按既有环境条件跳过；`git diff --check` 无内容错误，仅报告 Windows 工作区既有的 LF/CRLF 提示。Phase 2 继续进行，下一项为 P2.5“终局与战役复盘”。

## 当前路线 P2.5 终局与战役复盘

### 产品规则

- 正式结算同时冻结 `retrospective` v1 快照，不从继续沙盒后的世界状态重新推算旧结局。
- 主要势力明确标记为胜利、存续或败北流亡；流亡仍是可继续的故事状态，不是删除存档。

- 本地匿名漏斗增加创建、锁定、进入、3/6/9/12 月、战斗触发、完成、归档和继续沙盒节点，只接受字段白名单。

## 当前路线 P2.6 AI 基础目标

### 产品规则

- 固定首场战役锁定玩家时，主要 AI 势力立即公开一个 2～3 月目标，而不是等第一次月结后才显示。
- 目标来自实时资源、地理和威胁：稳粮、平乱、守边、祭祀扩军、蓄兵或夺取指定相邻城市。
- 同一目标在期限内持续存在；完成或到期后才重新选择，并保存前一目标及换目标原因。
- 目标影响既有 AI 行动优先级，但不提供资源、军令、战力或权限加成。突发缺粮/叛乱仍优先于普通目标。
- 中立城邦不获得主要 AI 的主动扩张目标，继续遵守“不受教唆不进攻”的独立规则。

### 实现结果

- `ai_goals.py` 持久化目标、历史、期限、进度和公开文案，并根据真实城市、路线、兵力、城防、叛乱、祭祀场和英灵数量选择目标。
- `ai.py` 优先执行与目标一致的合法方针、祭祀或指定夺城行动；不能满足条件时保留目标并解释受资源、军令或合法目标限制。
- 战情室新增“AI 战略动向”，展示原因、期限、进度条、上次行动和目标城市定位，不泄露尚未结算的随机战斗结果。
- 旧沙盒与中立城邦不启用主要 AI 目标层；新字段随世界 JSON 保存并兼容旧存档缺省值。

### 验证

- 战略单元测试覆盖两月目标持续、到期换目标、前一目标、缺粮方针、指定夺城、合法战斗和完成进度。
- HTTP 行为测试覆盖锁定时立即可见、跨月同一目标、目标城市、进度与行动说明。
- 前端行为测试覆盖 2～3 月说明、原因、进度条、上次行动和地图定位。
