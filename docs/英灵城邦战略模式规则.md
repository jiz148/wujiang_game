# 英灵城邦战略模式规则文档 v0.1

> 目标：本规则文档用于指导 Codex 实现一个长期沙盒战略模式。  
> 游戏已有基础：武将列表、多人/随机战斗、n*n 格子回合制战斗、武将技能/特性、武将 AI、手动操作与 AI 操作。  
> 新模式目标：在现有格子战斗系统外增加一个类似“三国志”玩法的战略层，但世界观不是三国，而是“英灵召唤 + 城邦战争 + 以太资源 + 世界危机事件”。

---

## 1. 模式定位

### 1.1 核心体验

玩家在一个由多个城市/节点组成的世界中扮演主公、武将或其他身份，经营势力、召唤英灵、管理城市、处理支持度、参与外交与战争，并在长期沙盒中经历世界主线事件、武将个人事件和动态故事事件。

核心乐趣是：

- 长期沙盒演化
- 城市经营与势力扩张
- 英灵召唤与圣物争夺
- 剧情事件与动态故事
- 战略层与格子战斗层结合
- 单人、合作、对战均可扩展

### 1.2 游戏结构

游戏由两层组成：

1. **战略层**
   - 城市经营
   - 资源管理
   - 官员任命
   - 圣物保管
   - 英灵召唤
   - 外交
   - 谋略
   - 行军
   - 围城
   - 事件推进

2. **战斗层**
   - 调用已有 n*n 格子回合制战斗系统
   - 玩家可手动操作
   - 也可让 AI 自动战斗
   - AI 自动战斗不是简单公式，而是双方 AI 在同一套战斗系统中实际模拟

## 2. 地图系统

### 2.1 地图类型

采用**抽象节点地图**。

节点可以是：

- 城市
- 关隘
- 遗迹
- 魔力矿脉
- 圣物地点
- 战场
- 中立村镇
- 世界事件地点

节点之间通过道路或连接线相连。

### 2.2 剧本结构

采用**随机地图随机势力**。

每局可以随机生成：

- 城市数量
- 城市资源分布
- 节点连接关系
- 初始势力
- 初始圣物位置
- 初始英灵线索
- 世界事件初始参数

后续可以扩展固定剧本，但 v0.1 以随机地图为主。

## 3. 玩家身份

战略模式采用正式职位控制。玩家和 AI 都不直接拥有势力权限，而是通过当前控制的职位执行操作：

```text
World -> Faction -> Office -> Controller(player | ai)
```

v0.1 固定职位仅包括：主公、大将军、将军、城主。谋士暂时仍是凡人职业或角色类型，不是独立职位；本阶段不加入大祭司、祭司、丞相、州牧或其他高级官职。

### 3.1 身份与目标

职位不改变战役最终目标，但决定玩家可见的工作台、可管理实体、可执行行动、义务、请求与报告。一个玩家可控制多个职位，但每次操作必须声明发起职位并通过该职位权限校验；不能在主公工作台直接执行城主或将军的底层操作。

### 3.2 正式职位系统

#### 3.2.0 玩家与 AI 的战略身份

玩家和战略 AI 的本体必须是一名来自现有武将池的唯一武将，不是账号化身、抽象势力控制器或无名官员。关系必须表示为：

```text
User / Hero AI -> StrategicHero -> Office (optional) -> Faction (optional)
```

- 一个玩家在同一战役中控制一名唯一武将；同一武将不能同时被多个玩家或 AI 控制。
- 武将可以担任主公，也可以处于在野状态。
- 在野武将可以选择建立新势力，成为新势力主公；也可以主动投靠已有主公。
- 已有势力必须由一名武将担任主公。不存在“账号直接成为主公”或“AI 直接控制势力”的情况。
- 玩家操作职位的前提，是其武将被正式任命为该职位；单人模式允许一名玩家切换自己武将兼任或临时接管的职位，但不能绕过武将身份。
- AI 同样以具体武将身份作出建国、投靠、任职、出征和离开势力等决定。

新战役开始时，玩家先选择一名尚未被占用的武将，再选择初始道路：

1. **成为主公**：接管一个可选初始势力并担任唯一主公。
2. **在野建国**：选择合法落脚城市建立新势力并成为主公。
3. **在野游历**：选择当前位置保持在野，后续可以投靠某位主公或在条件满足时建国。

旧存档迁移时，系统必须为每名原真人/AI 成员确定性分配一名未占用武将，并把原势力主公职位绑定到该武将，不能继续保留 `player_character` 或无名 AI 作为职位持有人。

#### 3.2.1 组织结构

```text
主公（每势力唯一）
├── 大将军（数量由国家科技容量决定）
│   └── 将军（必须隶属于一名大将军）
└── 城主（每座受控城市一个，直接隶属主公）
```

初始大将军基础容量为 1，每名大将军基础可辖 1 名将军。两类容量必须由配置计算，不写死在操作逻辑中：`基础容量 + 国家科技 office_capacity_delta`。军制改革分支增加大将军数量，参谋制度分支增加每名大将军可辖的将军数量。每个职位同一时间只有一个职位担任者，但控制者可为玩家或职位 AI。

#### 3.2.2 职位权限矩阵

| 行动领域 | 主公 | 大将军 | 将军 | 城主 |
|---|:---:|:---:|:---:|:---:|
| 任命/撤换大将军和城主 | 是 | 否 | 否 | 否 |
| 调整将军隶属/战区 | 是/批准 | 是（本战区） | 否 | 否 |
| 外交、宣战、停战、结盟 | 是 | 仅请求 | 否 | 否 |
| 国家科技、国家政策和预算 | 是 | 仅请求 | 否 | 否 |
| 在祭祀场举行召唤祭祀 | 是 | 否 | 否 | 是（仅限所辖城市） |
| 解除武将与祭祀场绑定 | 是 | 否 | 否 | 否 |
| 分配已仕官武将的职位与任务 | 是 | 否 | 否 | 否 |
| 从城市抽调已注册单位 | 否 | 是 | 仅请求直属大将军批准 | 否 |
| 增加本城兵力、注册士兵、规划建筑 | 否 | 否 | 否 | 是 |
| 发布国家战略目标 | 是 | 否 | 否 | 否 |
| 管理战区、军团目标和补给 | 仅发布目标/临时接管 | 是（所属战区） | 仅本军团 | 仅请求援助 |
| 行军、进攻、围城、撤退和战斗阵容 | 否 | 组织/分配 | 是（本军团） | 仅本地防御/请求 |
| 城市方针、资源、建设和本地守军 | 否 | 否 | 否 | 是（本城） |
| 本地叛乱和城市事件 | 势力危机/临时接管 | 仅军事援助 | 受命清剿 | 是（本城） |
| 月度战役结算 | 是（房主且控制主公） | 否 | 否 | 否 |

当前尚未实现的建筑、正式军团、外交和圣物操作仍保留为职位权限与工作台入口，不得由其他职位临时获得虚假的直接操作按钮。

#### 3.2.3 职位定义

- **主公**管理 `Faction`：国家科技、资源预算、外交、战争与和平、职位任命、国家目标、祭祀召唤和圣物决策。主公负责为全部已绑定武将分配职位和当前任务，也可以解除武将与祭祀场的绑定，但不直接调整城市方针、建筑、兵力或普通军团。主公可以亲自带兵出征；也可以向大将军下达进攻某城、守卫某城或建立防线的战略命令，但不能跳过大将军直接操作其他将军的军团。
- **大将军**管理 `Theater / Military Command`：所属将军、城市已注册单位的抽调、军团、战区补给、防线、集结点和军事目标。大将军可以把己方城市内确切的步兵、弓兵或骑兵单位调给直属将军；不能宣战、举行祭祀、研究国家科技或管理城市建筑。
- **将军**管理 `Army`：本军团路线、驻守、进攻、围城、撤退、支援、编制、补给、战斗控制和战报。将军不能直接从城市拿兵，必须向直属大将军提交包含城市、兵种和数量的调兵请求，经批准后单位才进入本军团。
- **城主**管理 `City`：城市方针、资源、建筑、学院、本地兵力、注册单位、城防、支持度、叛乱、本地事件和祭祀设施。城主可在所辖城市选择“增加兵力”或“注册士兵”，也可举行祭祀；不能任命、解除绑定或操作其他城市。

#### 3.2.4 权利、义务和命令链

每个职位必须同时保存 `permissions`、`duties`、`pending matters`。长期未完成义务可以降低支持度、补给或忠诚，并升级为危机。标准命令链为 `主公 -> 大将军 -> 将军 -> 军团`；城市命令链为 `主公 -> 城主 -> 城市执行`。上级发布目标，下级拆解执行，不能绕过中间职位直接操作底层实体。

职位之间使用结构化 `OfficeOrder` 传递命令、请求和报告。主公可向大将军发布“进攻目标城市 / 防守目标城市 / 建立防线”等军事目标，也可向城主发布发展目标；大将军再决定由哪个所属将军和军团执行。主公亲征时，由主公本人作为军队指挥武将，不需要虚构将军身份。城主、将军和大将军可向直属上级提交资源、援军、宣战或祭祀资源请求。

#### 3.2.5 空缺与临时接管

职位可处于 `active / vacant / disabled`。空缺城主仅执行基础自动管理且停止高级规划；空缺将军不能主动进攻；空缺大将军使所属将军协调效率下降；空缺主公触发摄政/继承危机。上级可通过 `OfficeTakeover` 临时接管空缺职责，但必须产生额外军令或管理负担，不能无限覆盖下级职位。

#### 3.2.6 数据模型

```yaml
Office:
  id: string
  faction_id: string
  office_type: enum[lord, grand_general, general, governor]
  holder_id: string | null
  holder_type: enum[hero, officer, player_character] | null
  controller_type: enum[player, ai]
  controller_user_id: int | null
  parent_office_id: string | null
  subordinate_office_ids: list[string]
  managed_entity_ids: list[string]
  permissions: list[string]

OfficeDuty:

OfficeOrder:
  id: string
  issuer_office_id: string
```

#### 3.2.7 独立职位工作台

- `LordWorkspace`：国家资源、城市概况、职位树与空缺、战区、国家科技、外交、圣物、可召唤英灵、世界主线、下属请求和势力待办。
- `GrandGeneralWorkspace`：所属战区地图、将军/军团、兵力补给、敌情、防线、集结点、主公命令、下属请求和军团调度。
- `GeneralWorkspace`：本军团位置、目标、路线、兵力、补给、士气、英灵、阵容、附近敌军、战斗方式、上级命令、援军/补给请求和战报。
- `GovernorWorkspace`：本城资源、人口、支持度、叛乱、建筑、学院、人才、守军、城防、祭坛、本地事件、主公目标和援助/召唤请求。

这些是不同的信息架构，不允许只在同一个全势力页面隐藏几个按钮来伪装职位差异。

## 4. 胜利与失败

### 4.1 胜利条件

游戏支持多种胜利条件并存。

可选胜利目标包括：

- 统一所有城市
- 消灭所有敌对势力
- 完成世界主线事件，例如解决雪鬼危机或电子危机
- 成为最大势力
- 收集一定数量的圣物
- 达成特定势力目标
- 完成身份相关目标

系统应允许多个胜利条件同时存在。

### 4.2 失败条件

游戏没有硬性失败。  
即使玩家失去主城、丢掉所有城市或大部分英灵，仍可继续以流亡、依附、重建等方式进行游戏。

可能的失败状态包括：

- 主城被攻陷
- 所有城市丢失
- 关键圣物被释放或夺走
- 势力支持度崩溃
- 世界事件失控
- 玩家势力灭亡

但这些状态不直接结束游戏，而是触发新的局面，例如：

- 流亡势力
- 投靠其他势力
- 作为武将继续游戏
- 从残余城市重建
- 进入世界危机后半段

### 4.3 结局评价

v0.1 不强制提供结局评价。  
玩家可以自己决定玩到什么时候结束。

后续可扩展：

- 历史年表
- 结局分支
- 统一评价
- 世界危机处理评价
- 英灵关系评价

## 5. 多人模式预留

多人模式统一使用职位控制，不再把玩家直接绑定为“整个势力控制者”。

### 5.1 不同势力模式

多个玩家可分别控制不同势力中的一个或多个职位。

适合：

- 对战
- 联盟
- 多势力沙盒
- 共同应对世界危机后再竞争

### 5.2 同一势力模式

多个玩家在同一势力中分别控制主公、大将军、将军或城主，并通过命令、请求和报告协作。

例如：

- 玩家 A 是主公
- 玩家 B 是大将军或将军
- 玩家 C 是城主
- 玩家 D 是另一座城市的城主

单人模式允许同一玩家控制多个职位并在工作台间切换；合作模式可以把职位分别分配给不同玩家。职位切换不改变权限，所有操作继续由当前工作台对应职位发起。

### 5.3 规则要求

多人和单人使用同一职位、命令链、职责与 AI 规则。玩家实体本身不授予势力权限。

## 6. 核心资源

每座城市拥有以下基础资源：

| 资源 | 作用 |
|---|---|
| 粮食 | 维持人口、驻军、远征、围城、支持度 |
| 金钱 | 建设、祭祀准备、外交、官员维护 |
| 人口 | 税收、征兵、城市发展 |
| 以太 | 召唤英灵、维护圣物、触发剧情 |
| 兵力 | 城市防守、出征、镇压、围城 |

### 6.1 以太

以太是唯一通用魔力资源。

以太用途：

- 召唤英灵
- 维护圣物
- 触发部分剧情、仪式、事件选择

以太不默认用于：

- 直接强化战斗
- 改变天气
- 升级普通建筑

以太应保持为“英灵/圣物/剧情”的专用核心资源。

## 7. 城市系统

### 7.1 城市属性

每座城市至少包含：

```yaml
City:
  id: string
  name: string
  owner_faction_id: string
  level: int
  food: int
  money: int
  population: int
  ether: int
  troops: int
  defense: int
  governor_id: string | null
  policies: list
  buildings: list
  relics_stored: list
  altars: list
  support_by_faction: dict[faction_id, int]
  local_factions: list
  traits: list
  event_states: list
```

### 7.2 城市支持度

每座城市对每个势力都有支持度，范围 0–100。

| 支持度 | 含义 |
|---:|---|
| 0–19 | 敌视 |
| 20–39 | 不满 |
| 40–59 | 观望 |
| 60–79 | 支持 |
| 80–100 | 拥护 |

支持度不等于城市归属。

一座城市可以属于玩家，但更支持另一个势力。  
支持度影响：

- 叛乱风险
- 收入效率
- 征兵效率
- 防守士气
- 敌方煽动成功率
- 投降概率
- 城主治理难度
- 事件触发

### 7.3 前端显示建议

后台可以记录城市对所有势力的支持度。  
前端只重点显示：

- 当前统治势力支持度
- 最高竞争势力支持度
- 自治/本地派系支持度
- 特殊势力影响，例如雪鬼、电子

避免 UI 信息过载。

## 8. 城主与城市委任

### 8.1 城主类型

城主可以是：

- 英灵武将
- 凡人官员

### 8.2 主城

主城也必须由对应城主负责具体经营，不能因为是主城就让主公直接规划普通建筑。主公负责国家政策、预算、职位任命、英灵召唤和圣物最终决策；主城城主负责把这些目标转化为城市建设、资源、学院、守军和本地事件操作。

### 8.3 非主城

非主城由对应城主工作台管理；若城主由 AI 控制，则职位 AI 根据主公目标和本地状态执行具体行动。

可用方针：

- 稳定优先
- 粮食优先
- 金钱优先
- 征兵优先
- 以太优先
- 城防优先
- 搜索优先
- 镇压优先
- 自治优先

### 8.4 非主城行动数量

非主城每月由城主执行 1–2 个行动。

建议：

| 城市等级 | 行动数量 |
|---|---:|
| 小城 | 1 |
| 中城 | 1，有概率追加辅助行动 |
| 大城 | 2 |
| 重镇/副都 | 2，可接受重点指令 |

## 9. 叛乱系统

### 9.1 叛乱风险

每座城市每月计算叛乱风险。

影响因素：

- 当前统治势力支持度低
- 粮食不足
- 税收过高
- 兵力不足
- 驻军过多
- 城主能力差
- 城主忠诚低
- 城主派系与城市冲突
- 敌方煽动
- 城市刚被攻占
- 雪鬼/电子等事件影响
- 当地派系活跃

### 9.2 叛乱阶段

叛乱分三阶段：

1. **隐患**
   - 只显示风险
   - 玩家可预防

2. **危机事件**
   - 弹出事件或进入日志
   - 玩家可安抚、镇压、调粮、谈判

3. **正式叛乱**
   - 城市进入叛乱状态
   - 可能发生战斗、倒戈或自治

### 9.3 叛乱类型

叛乱类型由城市派系、支持度结构和事件状态决定。

可能类型：

- 亲敌叛乱
- 自治叛乱
- 粮荒暴动
- 城主叛乱
- 反召唤派叛乱
- 电子派政变
- 雪鬼信徒暴动
- 圣物崇拜者叛乱

### 9.4 处理方式

玩家可通过以下方式处理：

- 派兵镇压
- 派英灵安抚
- 调粮赈济
- 减税
- 撤换城主
- 与派系谈判
- 调查敌方煽动
- 释放圣物
- 放弃城市

v0.1 已落地的主动处理动作：

- `appease` / 安抚民心：消耗势力金钱，提高目标城市统治支持度。
- `relief_grain` / 开仓赈济：消耗目标城市粮食，提高统治支持度。
- `suppress` / 派兵镇压：消耗目标城市兵力，提高短期秩序并压低本地自治派影响。
- 这些动作作为 `rebellion_action` 加入本月行动队列，房主推进月份时先于月度收入/维护/叛乱风险结算执行。
- 当月度叛乱风险达到正式叛乱阈值时，城市会生成或增长 `rebellion_force` 叛军状态，并造成守军、粮食、金钱和统治支持度损失；城市卡会显示叛乱风险和叛军规模。
- `suppress` 会优先削减现有 `rebellion_force`，叛军被清零时写入 `rebellion_suppressed` 事件。
- `rebellion_battle` / 清剿叛军：作为本月行动队列中的沙盒战斗动作，投入城市守军与 `rebellion_force` 结算；胜利清零叛军并写入 `rebellion_suppressed`，失败也会削弱叛军但造成守军与支持度损失。
- 城市卡在存在叛军时显示“计划清剿”按钮。
- 叛军真实格子战、倒戈/自治建国和谈判分支尚未落地。

## 10. 英灵武将系统

### 10.1 英灵定义

武将不是可直接购买或即时召唤的势力资源。无论其世界观来源是否与圣物或英灵殿有关，进入战略沙盒后，每名武将都是世界中唯一、持续存在并拥有位置、归属、忠诚和个人选择的政治主体。

### 10.1.1 在野、绑定与唯一获得方式

武将战略状态至少包括：

- `roaming`：在野，位于某座城市或地图节点，不属于任何势力。
- `serving`：已仕官，属于一个势力，可以担任职位或随军出征。
- `sleeping`：战败沉睡，保留原归属但暂时不能任职或参战。

取消“求贤、招募令、候选响应、城主举荐”整套设定。势力获得新武将的唯一正式方式是城市祭祀场举行的召唤祭祀：

1. 主公可选择任一己方城市举行祭祀；城主只能在自己所辖且建有祭祀场的城市举行祭祀。
2. 每次祭祀默认消耗 1 点势力军令和 30 点城市以太，从当前尚未绑定任何祭祀场的完整公开武将池中确定性随机召唤 1 名武将；未来新增到公开武将池的武将实时进入候选池。
3. 被召唤武将与召唤城市的祭祀场绑定，`ritual_city_id` 是其战略存在的保存位置。武将可离城任职或出征，但绑定位置不随移动改变。
4. 祭祀只能在势力仍有可用职位总容量时举行。容量按当前启用的非主公职位总数计算；已绑定本势力、尚未解除绑定的武将即使暂未任职，也占用一个容量。职位满员时必须先研究扩容科技、取得带来城主职位的新城市，或由主公解除一名武将绑定。
5. 解除绑定后，武将立即卸任、离开势力并恢复 `roaming`；其玩家/AI 武将身份仍保留，控制者可以继续在野行动。
6. 城市被敌方攻占时，该城市祭祀场保存的全部武将立即与原势力解除绑定并卸任，不能随祭祀场自动转投攻方。
7. 祭祀只改变绑定与势力归属，不自动授予职位；主公仍需用“任命武将”把无职位武将任命为大将军、将军或城主。
8. 在野武将建国时只获得本人担任的主公职位；其他职位保持空缺。旧存档中的求贤记录只作为历史数据读取，不再生成可执行操作。

每次祭祀不允许指定武将，也不允许在没有祭祀场、职位容量不足、以太不足或候选池为空时提交。

### 10.1.2 建筑、兵力与注册单位

城市建筑分为学院、田地、兵营、马厩、靶场和祭祀场，均有 1 至 3 级；建筑等级上限由主公研究的建筑科技分支决定。

- 学院：提供科技与建筑升级前置。
- 田地：提高每月粮食产出。
- 兵营：允许把城市兵力注册为步兵单位。
- 马厩：允许把城市兵力注册为骑兵单位。
- 靶场：允许把城市兵力注册为弓兵单位。
- 祭祀场：提高每月以太产出，并允许举行祭祀。

城主有两种互斥但可在不同军令中分别执行的兵务行动：

1. **增加兵力**：消耗人口、粮食和金钱，提高城市未注册兵力。
2. **注册士兵**：一次注册 1 至 3 个确切格子单位；候选兵种由本城建筑和本势力兵种科技共同决定，按战役种子、月份、城市和本次操作序号确定性随机组成。

注册消耗按照单位战力区分：步兵每单位消耗 100 城市兵力，弓兵每单位消耗 140，骑兵每单位消耗 180。已注册单位保存在 `City.registered_units`，不会再按百分比临时换算。大将军可把确切单位从己方城市调入直属将军的 `Office.unit_inventory`；普通将军必须提交结构化调兵请求，由直属大将军批准。将军发起城市战时优先使用自己持有的确切单位，城市防守则使用城内确切单位；不足部分才由旧存档兼容规则从未注册兵力换算普通步兵。

特点：

- 每个英灵拥有唯一圣物
- 召唤需要圣物、祭坛和以太
- 每月维护需要以太
- 战败不会永久死亡，而是沉睡
- 忠诚低时会拒绝命令
- 是格子战斗中的核心单位
- 拥有复杂技能、特性、AI、个人事件

### 10.2 英灵数据建议

```yaml
Hero:
  id: string
  name: string
  rank: enum[common, elite, famous, legendary, mythic]
  relic_id: string
  loyalty: int
  status: enum[not_summoned, active, sleeping, unavailable]
  sleep_remaining_months: int
  attributes: dict
  skills: list
  traits: list
  ai_profile: string
  relationships: dict[hero_id, int]
  personal_event_states: list
```

### 10.3 v0.1 城市战参战上限

每场城市战中，攻击方和防守方都可以投入已召唤且未沉睡的战略英灵。

- 默认每方每战最多投入 1 名英灵。
- 解锁战术科技 `英灵军议 / hero_command` 后，该势力每场城市战英灵投入上限 +1。
- 攻击方在发起即时进攻或提交月度进攻计划时选择参战英灵。
- 防守方可以设置默认防守英灵；也可以在单场 pending battle 尚未创建真实格子战房间前设置本场防守英灵。
- 真实格子战房间创建后，本场参战英灵名单锁定，不再允许修改。

### 10.4 战败沉睡

英灵战败后进入沉睡。  
沉睡时间由等级决定。

| 英灵等级 | 沉睡时间 |
|---|---:|
| 普通 | 1 个月 |
| 精英 | 2 个月 |
| 名将 | 3 个月 |
| 传说 | 4–6 个月 |
| 神话 | 6 个月以上或需要特殊事件唤醒 |

### 10.5 忠诚度

英灵忠诚低不会直接投敌。  
但可能：

- 拒绝出战
- 拒绝镇守
- 拒绝镇压
- 拒绝谋略
- 触发质问事件
- 要求释放某圣物
- 不接受玩家手动/AI 指令而按自身性格行动

忠诚层级：

| 忠诚 | 状态 | 效果 |
|---:|---|---|
| 80–100 | 信任 | 正常行动，可能触发正面事件 |
| 60–79 | 稳定 | 正常行动 |
| 40–59 | 疏离 | 可能拒绝部分命令 |
| 20–39 | 抗拒 | 经常拒绝命令 |
| 0–19 | 决裂 | 基本无法有效指挥 |

## 11. 圣物系统

### 11.1 圣物定义

每个英灵绑定一个唯一圣物。  
圣物用于召唤和维持对应英灵。

### 11.2 圣物状态

```yaml
Relic:
  id: string
  name: string
  hero_id: string
  state: enum[scattered, stored, bound_to_altar, released]
  location_id: string | null
  owner_faction_id: string | null
  altar_id: string | null
  maintenance_ether_cost: int
```

圣物状态说明：

1. **散落**
   - 位于世界某处
   - 需要搜索、事件、外交或战争获得

2. **保管**
   - 被某势力拥有
   - 存放在城市、圣物库或队伍中
   - 城市失守时可能被夺

3. **祭坛绑定**
   - 放入祭坛，可召唤英灵
   - 无法正常取下
   - 取下等于释放

4. **释放**
   - 主动断开控制
   - 圣物重新随机/半随机分布到世界某处
   - 防止敌人夺取

### 11.3 圣物夺取

如果城市失守：

- 保管状态圣物：被胜方夺取
- 祭坛绑定圣物：若未提前释放，被胜方夺取
- 已释放圣物：不会被夺取，重新散落

### 11.4 圣物释放

玩家或 AI 可主动释放圣物。

释放后：

- 圣物离开当前城市/祭坛
- 对应英灵无法继续由当前势力召唤
- 圣物重新分布到世界某处
- 可能触发英灵个人事件
- 可能生成传闻或线索

建议采用半随机分布：

- 可能回到英灵关联地区
- 可能回到遗迹
- 可能出现在中立城市
- 可能进入世界事件区域

## 13. 凡人官员系统

### 13.1 凡人官员定义

凡人官员是非英灵的人类角色，可担任：

- 城主
- 谋士
- 将领

特点：

- 不需要圣物
- 不需要以太维护
- 会老去、死亡、阵亡
- 有忠诚度
- 有派系
- 可成长为优秀官员，但上限低于英灵
- 负责国家日常运转

### 13.2 凡人与英灵区别

| 项目 | 英灵 | 凡人 |
|---|---|---|
| 来源 | 圣物召唤 | 搜索、学院、事件 |
| 需要圣物 | 是 | 否 |
| 需要以太 | 是 | 否 |
| 永久死亡 | 否，沉睡 | 是 |
| 战斗定位 | 核心单位 | 辅助、统率、战略层 |
| 忠诚低 | 拒绝命令 | 离职、投敌、倒戈 |
| 成长上限 | 高 | 低于英灵 |
| 事件复杂度 | 高 | 中低 |

### 13.3 凡人来源

学院可以增加年轻人才出现概率。  
年轻人才需要通过搜索行动发现。

人才来源：

- 城市自然出现
- 学院培养
- 搜索发现
- 事件获得
- 失势旧臣
- 地方豪族
- 流浪将领
- 特殊剧情人物

### 13.4 学院

学院效果：

- 提高年轻人才出现概率
- 提高优秀人才概率
- 提高搜索成功率
- 提供官员培训
- 降低官员断层风险
- 触发学院事件

### 13.5 寿命与死亡

凡人有年龄和寿命。

年龄阶段：

- 青年：成长快，能力低
- 壮年：能力稳定
- 老年：经验高，行动效率可能下降
- 高龄：可能退隐、病亡、推荐弟子

凡人在战败时可能：

- 逃脱
- 受伤
- 被俘
- 投降
- 阵亡

### 13.6 忠诚

凡人忠诚低可能：

- 懈怠
- 贪污
- 泄密
- 离职
- 投敌
- 带兵倒戈
- 发动地方叛乱

每个凡人官员都有派系。

## 14. 每月行动系统

### 14.1 行动资源

每月有以下行动资源：

1. 主城行动
2. 英灵行动
3. 凡人官员行动
4. 非主城委任行动
5. 祭坛行动

#### 14.1.1 v0.1 势力军令

每个势力每月共享 **4 点势力军令**，但每条军令必须记录 `issuer_office_id` 并通过职位权限校验。多人在同一势力共享预算；AI 职位也使用同一总量。每座城市仍最多接受 2 条由该城城主或合法临时接管职位发起的城市军令。

| 行动 | 军令消耗 |
|---|---:|
| 调整城市方针 | 1 |
| 解锁战术科技 | 1 |
| 召唤英灵 | 1 |
| 安抚/赈济叛乱 | 1 |
| 派兵镇压/清剿叛军 | 2 |
| 发动城市进攻 | 2 |
| 流亡行动 | 1 |

同一行动键的计划可在月结前替换，替换只按新计划重新计算占用，不重复消耗军令。军令在进入下个月时自动恢复。

每月公开状态还应为每个势力生成一份基于当前世界状态的局势简报，至少包括当前最大威胁、当前最佳机会和一个带有不确定性提示的敌方动向推测。局势简报用于帮助玩家快速理解地图，不直接修改世界状态，也不保证敌情推测完全准确。

### 14.2 主城行动

主城行动数由主城建筑等级决定。

| 主城等级 | 每月主城行动 |
|---|---:|
| 1 | 1 |
| 2 | 2 |
| 3 | 3 |
| 4 | 4 |
| 王都/圣都 | 5 |

主城行动可用于：

- 建设
- 升级
- 建立祭坛
- 建立圣物库
- 大规模征兵
- 大规模赈济
- 制定政策
- 强化防御
- 迁都/设副都

### 14.3 英灵行动

每个可行动英灵每月有 1 次行动。

可执行：

- 出征
- 驻守
- 训练
- 搜索圣物
- 安抚城市
- 执行谋略
- 镇压叛乱
- 参与剧情事件
- 拜访其他英灵
- 修复关系

### 14.4 凡人官员行动

凡人每月通常有 1 次行动。

按职位可执行：

| 职位 | 行动 |
|---|---|
| 城主 | 治理、征粮、建设、训练守军、安抚 |
| 谋士 | 谋略、调查、煽动、反间、反制 |
| 将领 | 领兵、守城、镇压、护送、训练 |

### 14.5 外交与谋略

外交不消耗行动资源。

外交可能消耗：

- 金钱
- 粮食
- 以太
- 圣物承诺
- 条约信用
- 关系

外交应有冷却和失败代价，避免无限刷外交。

谋略消耗英灵或谋士行动点。

谋略包括：

- 煽动叛乱
- 离间城主
- 破坏粮仓
- 调查圣物
- 刺探军情
- 反制敌方谋略
- 暗中提高己方城市支持度

## 16. 战斗接入系统

### 16.1 兵力转化

普通兵力进入格子战斗时转化为普通士兵单位。

建议：

```text
100 兵力 = 1 个普通士兵单位
```

每方普通士兵单位应有上限，例如 12 个。  
超过上限的兵力作为后备、士气或援军波次处理。

### 16.2 士兵类型

- 普通士兵

- 以太兵

   - 播放过程
   - 玩家可观看

   - 输出战报

### 19.6 事件定义建议

```yaml
EventDefinition:
  id: string
  name: string
  category: enum[world_main, world_side, city, character, dynamic_story]
  visibility: enum[player_visible, rumor, background_only]
  trigger_type: enum[fixed_month, condition, weighted_random, battle_result, support_change, relic_state_change, memory_tag]
  trigger_conditions: dict
  weight: int
  cooldown_months: int
  choices: list
  effects: list
  memory_tags_added: list
  memory_tags_removed: list
```

### 19.7 v0.1 连续事件闭环

每个非流亡势力同时最多拥有 1 个待决事件。事件在月初出现，并包含 2 至 3 个可主动选择的处理方式：

- 主动选择进入本月行动队列，消耗 1 点势力军令。
- 同一事件的选择可以在月结前替换，不重复占用军令。
- 选项可以要求并消耗城市或势力资源；资源不足时不可选择。
- 月底仍未处理的事件自动采用隐藏的“放任”结果，不消耗军令，但通常产生负面影响。
- 部分选择会创建 1 至 2 个月后结算的延迟后果，并在界面显示为“未完影响”。
- 选择与延迟后果会写入世界记忆标签，供后续事件链继续引用。

v0.1 首批事件链包括：边境逃兵、饥民请愿、以太异象、行会争端。每个势力的事件由战役种子、月份和势力稳定生成，并避免连续三个月重复同一模板。AI 势力使用同一选项与资源规则，并从自己的 4 点军令中支付事件选择费用。

## 21. AI 势力系统

### 21.1 AI 与玩家规则一致

AI 完全使用和玩家相同的规则。

AI 不再作为全能势力控制器直接执行动作。势力级协调器只能产生目标，实际行动必须委派给 `LordOfficeAI / GrandGeneralOfficeAI / GeneralOfficeAI / GovernorOfficeAI`，并通过与玩家相同的职位权限、管理对象、军令和资源检查。

AI 也拥有：

- 城市
- 资源
- 官员
- 英灵
- 圣物
- 祭坛
- 支持度
- 忠诚度
- 外交
- 谋略
- 军队
- 事件记忆

### 21.2 AI 每月决策

AI 每月综合考虑：

- 扩张
- 稳定
- 圣物寻找
- 英灵召唤
- 资源管理
- 城市支持度
- 叛乱处理
- 世界事件
- 外交
- 战争

v0.1 已落地的 AI 边界：

- AI 控制未被真人战役成员占用的势力；房主锁定初始玩家时，所有空置初始势力会转为 AI 托管。
- 战役 resume gate 只要求真人初始玩家在线，AI 托管势力不需要在线。
- 已流亡势力不执行普通城市经营/出征 AI。
- 房主推进月份时，先结算玩家月度队列，再执行 AI 月度计划，最后进入月度资源/维护/叛乱结算。
- AI 主公职位负责国家科技、英灵召唤和势力事件；AI 城主负责本城方针、资源、叛乱和本地事件；AI 大将军负责战区目标和协调；AI 将军负责具体进攻、战斗方式和阵容。每项行动日志必须记录发起职位。
- AI 城市方针优先级必须把生存风险放在普通发展前面：缺粮城市先切粮食；叛乱风险高且有足够守军时切镇压；叛乱风险高但守军不足时切自治；中高叛乱风险且安全兵力不足时先征兵。
- AI 发动 quick 进攻时，会投入当前可用战略英灵，数量受该势力战略英灵参战上限限制。
- AI 行动写入 `strategy_ai_plan` 事件日志，方便玩家追踪和测试回归。
- 外交、圣物释放、官员任命、英灵任务、叛乱事件链、AI 主动叛乱处理、叛军战斗和世界主线压力仍属于后续扩展。

### 21.3 AI 外交

AI 会主动外交，包括：

- 结盟
- 停战
- 互不侵犯
- 共同讨伐
- 交换资源

### 21.4 AI 释放圣物

触发判断包括：

- 城市被围
- 失守概率高
- 圣物价值高
- 无援军
- 圣物被敌人夺取后后果严重

### 21.5 AI 没有固定性格

所有 AI 使用统一决策逻辑。  
不同势力的行为差异来自：

- 地理
- 资源
- 城市支持度
- 英灵构成
- 圣物状态
- 世界事件压力
- 官员能力
- 外交关系

### 21.6 AI 战斗

重要战斗真实模拟。  
小战快速结算。

重要战斗包括：

- 玩家可见区域战斗
- 主城战

## 24. 关键设计原则

1. 英灵是传奇核心，不要让普通士兵和凡人官员抢走战斗焦点。
2. 凡人官员是国家骨架，用来支撑城市管理和长期沙盒。
3. 以太只服务英灵、圣物和剧情，避免变成普通资源。
4. 圣物争夺是战略核心之一。
5. 城市支持度要让城市有政治生命，而不是单纯资源点。
6. 自动战斗应尽量调用真实格子战斗 AI 模拟。
7. 事件系统是模式灵魂，应能改变世界。
8. AI 与玩家同规则，保证沙盒可信。
9. 没有硬失败，失败也要转化为新故事。
10. 多人模式通过权限分配接入，不改变核心规则。
